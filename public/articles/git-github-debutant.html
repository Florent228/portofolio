01 —Pourquoi Git et GitHub sont incontournables

Imaginez que vous travaillez sur un document important. Vous le modifiez, vous le modifiez encore, et un jour vous réalisez que vous avez tout cassé. Comment revenir en arrière ? Si votre seule sauvegarde est document_final_v3_VRAIMENT_FINAL.docx, vous avez un problème.

Git résout exactement ça, mais pour du code. C'est un système de contrôle de version : il enregistre chaque modification de votre projet, qui l'a faite, quand, et pourquoi. Vous pouvez voyager dans le temps, comparer des versions, et revenir à n'importe quel état passé.

Pourquoi c'est critique

En 2024, plus de 100 millions de développeurs utilisent GitHub. La quasi-totalité des entreprises technologiques exigent Git comme compétence de base. Ce n'est pas optionnel : c'est le standard universel du développement logiciel.

Sans Git Avec Git
Sauvegarder manuellement des copies (v1, v2, final…) ✓ Historique complet et automatique de chaque modification
Impossible de travailler à plusieurs sans écraser le travail des autres ✓ Collaboration simultanée sur le même projet, sans conflits
En cas de bug, trouver la cause prend des heures ✓ Identifier en quelques secondes quel commit a introduit le bug
Tester une nouvelle fonctionnalité = risquer de tout casser ✓ Les branches isolent les expériences du code stable

Et GitHub, c'est quoi alors ? C'est une plateforme en ligne qui héberge vos dépôts Git. Git est l'outil local sur votre machine. GitHub est le cloud où vous synchronisez, partagez et collaborez. Ce sont deux choses distinctes — Git peut s'utiliser sans GitHub, mais le combo est aujourd'hui le standard.

02 —Les concepts fondamentaux (sans jargon)

Avant de taper la moindre commande, il faut comprendre quelques notions. Ne vous inquiétez pas : c'est plus simple que ça n'en a l'air.

Le dépôt (repository ou "repo")

C'est le dossier de votre projet, augmenté d'un sous-dossier caché .git/ qui contient tout l'historique. Un repo peut être local (sur votre machine) ou distant (sur GitHub).

Le commit

Un commit est une photo instantanée de votre projet à un moment donné. Chaque commit a un message qui explique ce qui a changé, un auteur, et une date. L'ensemble des commits forme une ligne du temps que vous pouvez parcourir à tout moment.

💡 Bonne pratique

Un bon message de commit répond à : "Si appliqué, ce commit va…" Par exemple : "Ajoute la validation du formulaire de connexion" plutôt que "modifs" ou "fix".

La branche (branch)

Imaginez l'historique de votre projet comme une rivière. Une branche est un affluent : vous pouvez y travailler de façon isolée, puis fusionner avec la rivière principale (main) quand c'est prêt. La branche main (ou master) est la version stable de votre projet.

Le cycle de vie d'un fichier

WD
Zone de travail
Working Directory
SA
Zone de transit
Staging Area
GIT
Dépôt local
Local repo
GH
GitHub
Remote repo

Vous modifiez vos fichiers dans le Working Directory. Vous choisissez quels changements inclure dans le prochain commit avec git add (Staging Area). Vous validez le commit avec git commit. Vous envoyez vers GitHub avec git push.

03 —Installation et configuration

Installer Git

Git est disponible sur tous les systèmes d'exploitation. La méthode la plus simple selon votre OS :

// macOS

# Via Homebrew (recommandé) brew install git # Ou téléchargez depuis : https://git-scm.com/download/mac

// Windows

# Téléchargez Git for Windows : https://git-scm.com/download/win # Ou via winget : winget install --id Git.Git -e --source winget

// Linux (Ubuntu/Debian)

sudo apt update && sudo apt install git

Configuration initiale (obligatoire)

Après l'installation, identifiez-vous. Git a besoin de savoir qui fait les commits :

// Terminal — Configuration globale

git config --global user.name "Votre Nom" git config --global user.email "votre@email.com" # Vérifier la configuration git config --list

ℹ️ Note

Utilisez la même adresse e-mail que votre compte GitHub. C'est ce qui permet d'associer vos commits à votre profil sur la plateforme.

04 —Votre premier dépôt en 10 minutes

Passons à la pratique. Voici comment créer un projet, enregistrer des modifications et pousser sur GitHub, étape par étape.

Étape 1 — Créer un dépôt local

// Terminal

# Créer un dossier et l'initialiser comme dépôt Git mkdir mon-premier-projet cd mon-premier-projet git init Initialized empty Git repository in /mon-premier-projet/.git/

Étape 2 — Créer un fichier et faire un commit

// Terminal

# Créer un fichier echo "# Mon Premier Projet" > README.md # Voir l'état du dépôt git status On branch main Untracked files: README.md # Ajouter le fichier au staging git add README.md # Créer le commit git commit -m "Ajoute le fichier README initial" [main (root-commit) a3f82c1] Ajoute le fichier README initial

Étape 3 — Connecter à GitHub et pousser

Sur GitHub, créez un nouveau dépôt (bouton New repository), ne cochez rien (ni README ni .gitignore), et copiez l'URL fournie.

// Terminal — Connecter et envoyer

# Ajouter GitHub comme dépôt distant git remote add origin https://github.com/votreuser/mon-projet.git # Envoyer (pousser) votre code sur GitHub git push -u origin main Branch 'main' set up to track remote branch 'main' from 'origin'. Everything up-to-date

✅ Félicitations

Votre code est maintenant sur GitHub. Refreshissez la page GitHub — vous verrez votre README affiché. Après cette première configuration, un simple git push suffit pour envoyer vos modifications.

05 —Travailler avec les branches

Les branches sont l'une des fonctionnalités les plus puissantes de Git. Elles permettent de tester, expérimenter, casser des choses — sans jamais mettre en danger le code qui fonctionne.

La règle d'or : ne jamais travailler directement sur main. Créez toujours une branche pour chaque fonctionnalité ou correction.

// Terminal — Cycle de vie d'une branche

# Créer et basculer sur une nouvelle branche git switch -c feature/ajout-formulaire # Travailler, modifier des fichiers, puis committer normalement git add . git commit -m "Ajoute le formulaire de contact" # Revenir sur main et fusionner votre branche git switch main git merge feature/ajout-formulaire # Supprimer la branche une fois fusionnée (bon ménage) git branch -d feature/ajout-formulaire

⚠️ Les conflits de fusion

Si deux personnes modifient la même ligne d'un fichier, Git crée un conflit. Ne paniquez pas : Git marque les zones conflictuelles dans le fichier, vous choisissez quelle version garder, vous commitez. C'est une compétence qui s'apprend en pratiquant.

06 —Collaborer sur GitHub

Le fork

Un fork est une copie d'un dépôt appartenant à quelqu'un d'autre dans votre propre compte GitHub. C'est la base de la contribution open source : vous forkez, vous modifiez, vous proposez vos changements.

La Pull Request (PR)

Une Pull Request est une proposition de fusion. Vous dites à l'équipe : "J'ai ce code sur ma branche, est-ce que vous voulez bien l'intégrer dans main ?" C'est là que la revue de code (code review) a lieu, où les collègues commentent, suggèrent des améliorations, et approuvent.

ℹ️ Workflow typique en équipe

1. Tirer les dernières modifications : git pull
2. Créer une branche : git switch -c feature/ma-feature
3. Coder, committer régulièrement
4. Pousser la branche : git push origin feature/ma-feature
5. Ouvrir une Pull Request sur GitHub
6. Faire la revue de code, corriger si besoin
7. Fusion (merge) dans main

Récupérer les changements des collègues

// Terminal — Synchronisation

# Récupérer ET fusionner les changements distants git pull # Récupérer SANS fusionner (pour voir d'abord) git fetch # Cloner un dépôt existant (première fois) git clone https://github.com/user/projet.git

07 —Mémo des commandes essentielles

Voici les commandes que vous utiliserez chaque jour. Gardez cette page en marque-page.

git init

Initialise un nouveau dépôt Git dans le dossier courant

git clone <url>

Copie un dépôt distant sur votre machine

git status

Affiche l'état actuel des fichiers (modifiés, en staging, etc.)

git add .

Ajoute tous les fichiers modifiés au staging

git commit -m "msg"

Enregistre les changements en staging avec un message

git push

Envoie les commits locaux vers GitHub

git pull

Récupère et fusionne les changements distants

git log --oneline

Affiche l'historique des commits en résumé

git switch -c <nom>

Crée et bascule sur une nouvelle branche

git merge <branche>

Fusionne une branche dans la branche courante

git diff

Montre les modifications non encore commitées

git stash

Met de côté les modifications en cours sans committer

08 —Les erreurs fréquentes (et comment s'en sortir)

"J'ai commité sur main par erreur"

// Solution

# Annuler le dernier commit (garde les modifications) git reset --soft HEAD~1 # Puis créer votre branche et recommitter git switch -c feature/ma-branche git commit -m "Mon vrai message"

"J'ai écrit un mauvais message de commit"

// Solution (uniquement si pas encore poussé)

git commit --amend -m "Le bon message corrigé"

"Je veux ignorer certains fichiers"

Créez un fichier .gitignore à la racine du projet et listez les fichiers/dossiers à exclure :

// .gitignore

# Dépendances Node.js node_modules/ # Variables d'environnement (ne JAMAIS commiter !) .env .env.local # Fichiers système .DS_Store Thumbs.db # Builds dist/ build/

⚠️ Règle absolue

N'envoyez jamais de mots de passe, clés API ou fichiers .env sur GitHub — même sur un dépôt privé. Utilisez toujours .gitignore et des services comme GitHub Secrets pour les données sensibles.

La suite de votre apprentissage

Une fois ces bases maîtrisées, explorez dans cet ordre : le rebase interactif pour nettoyer l'historique, les GitHub Actions pour automatiser vos tests et déploiements, les tags et releases pour versionner votre logiciel, et les stratégies de branches comme Git Flow ou trunk-based development.

🚀 La clé : pratiquer tous les jours

La meilleure façon d'apprendre Git, c'est de l'utiliser sur de vrais projets. Créez des repos pour vos exercices, contribuez à de l'open source sur GitHub, et consultez git help <commande> sans hésiter. En deux semaines d'usage quotidien, tout deviendra instinctif.
